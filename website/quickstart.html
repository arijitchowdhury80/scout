Scout Quickstart

Use hosted Scout with an API key, or run it locally.

If you are using this website, start with the hosted API examples below. Install locally when you want your own workdir, local artifacts, or browser/session capture on your machine.

Path 01 · Hosted beta

Call the live Scout API.

Hosted beta is invite-only and metered. Use the domain below with the API key delivered to you. Do not use localhost for hosted calls. 

export SCOUT_HOSTED_BASE_URL="https://scout.chowmes.com"
export SCOUT_HOSTED_API_KEY="paste-your-delivered-key"

curl "$SCOUT_HOSTED_BASE_URL/v1/hosted/me" \
  -H "Authorization: Bearer $SCOUT_HOSTED_API_KEY"

curl -X POST "$SCOUT_HOSTED_BASE_URL/v1/hosted/scrape" \
  -H "Authorization: Bearer $SCOUT_HOSTED_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"url":"https://example.com","formats":["markdown"],"timeout_ms":30000}'

Use the real domain

Hosted calls go to `https://scout.chowmes.com/v1/hosted/*` with `Authorization: Bearer ...`.

Credits

`/v1/hosted/me` shows your plan, balance, and current limits. A basic hosted scrape consumes one standard credit.

Keys

Never paste hosted API keys into frontend code, public repos, screenshots, tickets, or shared logs.

Path 02 · Local install

Run Scout on your own machine.

python3 -m venv .venv
source .venv/bin/activate
pip install "git+https://github.com/arijitchowdhury80/scout.git@codex/scout-platform-foundation"
playwright install chromium
export SCOUT_WORKDIR="$PWD/scout-runs"
scout serve

Workdir

`SCOUT_WORKDIR` is where local Scout writes records, source pages, blocked pages, validation, reports, raw evidence, and screenshots.

Local health check

Only after `scout serve` is running locally, open `http://127.0.0.1:8421/health` or `http://127.0.0.1:8421/docs` on the same machine.

Package release

`pip install scout-web` comes after registry publishing gates close. Until then, beta installs use the verified launch branch.

Path 03 · Docker

Use Docker when you want a clean local service boundary.

docker compose -f docker/docker-compose.yml up -d
curl http://127.0.0.1:8421/health

Persistence

Mount a host volume for Scout runs so artifacts survive container restarts.

Browser capture

Docker is best for service-style scraping. User-browser capture remains a local-machine workflow because it can touch your active session.

First useful run

Make one artifact folder, then inspect it.

1. Pick a target

Start with a public page you are allowed to access.

2. Run Scout

Use CLI or HTTP API. Keep live hard-site testing small and intentional.

3. Inspect evidence

Open `records.json`, `source_pages.json`, `blocked_pages.json`, and `extraction_report.md`.

Readiness check

Verify the same launch gate CI runs.

The readiness checker summarizes the current truth: the controlled beta release is ready with limits. Public self-serve SaaS, paid checkout, public registries, and security-clean claims remain future gates. 

scout launch-readiness
scout launch-readiness --json
scout launch-readiness --require-public

Beta gate

The default command should report `Private beta: ready_with_limits` when the repo evidence is intact.

Release gate

`--require-public` should pass for the current controlled beta release.
