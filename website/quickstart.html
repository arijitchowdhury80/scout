Path 01 · Hosted beta

Call the live Scout API.

Hosted beta is invite-only and metered. Use the domain below with the API key delivered to you. Do not use localhost for hosted calls. Hosted credits are capped while unit economics are validated; Scout does not offer unlimited hosted crawling. 

export SCOUT_HOSTED_BASE_URL="https://scout.chowmes.com"
export SCOUT_HOSTED_API_KEY="paste-your-delivered-key"

curl "$SCOUT_HOSTED_BASE_URL/v1/hosted/me" \
  -H "Authorization: Bearer $SCOUT_HOSTED_API_KEY"

curl -X POST "$SCOUT_HOSTED_BASE_URL/v1/hosted/scrape" \
  -H "Authorization: Bearer $SCOUT_HOSTED_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"url":"https://example.com","formats":["markdown"],"timeout_ms":30000}'

Use the real domain

Hosted calls go to `https://scout.chowmes.com/v1/hosted/*` with `Authorization: Bearer ...`.

Credits

`/v1/hosted/me` shows your plan, balance, and current limits. A basic hosted scrape consumes one standard credit.

Keys

Never paste hosted API keys into frontend code, public repos, screenshots, tickets, or shared logs.

API reference

The endpoints testers actually need.

GET /v1/hosted/me Check hosted beta plan, credits, limits, and retention.
POST /v1/hosted/scrape Acquire one public URL and return content plus evidence metadata.
GET /health Verify a local or hosted Scout service is alive.
POST /scrape Local authenticated scrape route for agent and workflow integrations.
POST /products Local product acquisition route that writes product artifacts.
POST /run/{use_case} Local intelligence runner for company, PRISM, investor, careers, jobs, news, research, docs, social, locations, and website-quality workflows.

Path 02 · Local install

Run Scout on your own machine.

python3 -m venv .venv
source .venv/bin/activate
pip install "git+https://github.com/arijitchowdhury80/scout.git@codex/scout-platform-foundation"
playwright install chromium
export SCOUT_WORKDIR="$PWD/scout-runs"
scout serve

Workdir

`SCOUT_WORKDIR` is where local Scout writes records, source pages, blocked pages, validation, reports, raw evidence, and screenshots.

Local-only health check

Only after `scout serve` is running on your own machine, open the local health or Swagger page from that same machine.

Package release

`pip install scout-web` comes after registry publishing gates close. Until then, beta installs use the verified launch branch.

Path 03 · Docker

Use Docker when you want a clean local service boundary.

docker compose -f docker/docker-compose.yml up -d
curl http://localhost:8421/health

Persistence

Mount a host volume for Scout runs so artifacts survive container restarts.

Browser capture

Docker is best for service-style scraping. User-browser capture remains a local-machine workflow because it can touch your active session.

First useful run

Make one artifact folder, then inspect it.

1. Pick a target

Start with a public page you are allowed to access.

2. Run Scout

Use CLI or HTTP API. Keep live hard-site testing small and intentional.

3. Inspect evidence

Open `records.json`, `source_pages.json`, `blocked_pages.json`, and `extraction_report.md`.

Examples

Run small, inspectable jobs first.

Page to markdown

Capture one public page as readable text with source metadata.

mkdir -p ./scout-runs/example-page
scout scrape https://example.com \
  > ./scout-runs/example-page/scrape.json

Product category to records

Create product records plus blocked-page evidence for a small listing run.

scout products books \
  --site books.toscrape.com \
  --start-url https://books.toscrape.com/ \
  --output-dir ./scout-runs/books-products

Company intelligence packet

Produce a compact company packet with overview, key URLs, and citable source pages.

scout run company \
  --query Adobe \
  --mode auto \
  --workdir ./scout-runs/adobe-company

Artifact contract

Every useful run should leave evidence behind.

records.json

Structured records created from the run.

source_pages.json

Source registry with URLs, providers, timestamps, and hashes.

blocked_pages.json

Blocked, sparse, or failed pages with provider attempts and reasons where available.

extraction_report.md

A human-readable summary of what worked, what failed, and what should be reviewed next.

Readiness check

Verify the same launch gate CI runs.

The readiness checker summarizes the current truth: the controlled beta release is ready with limits. Public self-serve SaaS, paid checkout, public registries, and security-clean claims remain future gates. 

scout launch-readiness
scout launch-readiness --json
scout launch-readiness --require-public

Beta gate

The default command should report `Private beta: ready_with_limits` when the repo evidence is intact.

Release gate

`--require-public` should pass for the current controlled beta release.
