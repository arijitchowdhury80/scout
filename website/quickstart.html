Path 01 · Hosted beta

Call the live Scout API.

Hosted HTTP beta starts through `/beta` and is always metered. When the hosted stack is ready, the beta form starts with $0 card-backed setup so Stripe Checkout and webhook provisioning are exercised. Email-only registration remains available as a fallback request queue. Do not use localhost for hosted calls. Hosted credits are capped while unit economics are validated; Scout does not offer unlimited hosted crawling. 

export SCOUT_HOSTED_BASE_URL="https://scout.chowmes.com"
export SCOUT_HOSTED_API_KEY="paste-your-delivered-key"

curl "$SCOUT_HOSTED_BASE_URL/v1/hosted/me" \
  -H "Authorization: Bearer $SCOUT_HOSTED_API_KEY"

curl -X POST "$SCOUT_HOSTED_BASE_URL/v1/hosted/scrape" \
  -H "Authorization: Bearer $SCOUT_HOSTED_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"url":"https://example.com","formats":["markdown"],"timeout_ms":30000}'

Private beta access

Register for your beta tester API key.

Beta testers start with $0 card-backed setup when available. Scout records the tester, provisions a finite-credit beta account after signed checkout completion, and emails the API key. Public signup never shows raw keys in the browser.

Get beta tester API key

Use the hosted beta form for $0 setup or email-only fallback. Existing keys still work with the hosted API examples below.

Already have a key? Open Hosted Account to inspect credits, recent usage, and purchase history without creating a login.

Use the real domain

Hosted calls go to `https://scout.chowmes.com/v1/hosted/*` with `Authorization: Bearer ...`.

Credits

`/v1/hosted/me` shows your plan, balance, and current limits. A basic hosted scrape consumes one standard credit.

Keys

Recommended beta onboarding starts at `/v1/billing/stripe/checkout-session` with package `beta_trial`, your name, and your email. The fallback request queue uses `/v1/hosted/beta-key`. Scout emails the API key; never paste hosted API keys into frontend code, public repos, screenshots, tickets, or shared logs.

API reference

The endpoints testers actually need.

GET /v1/hosted/me Check hosted beta plan, credits, limits, and retention.
POST /v1/hosted/beta-key Register for a finite-credit beta tester API key by email.
POST /v1/hosted/scrape Acquire one public URL and return content plus evidence metadata.
GET /health Verify a local or hosted Scout service is alive.
POST /v1/hosted/products Acquire product records through the hosted beta surface.
POST /v1/hosted/run/{use_case} Run company, PRISM, investor, careers, news, social, and locations workflows.
GET /v1/hosted/runs List hosted run summaries owned by the caller's beta key.

Operator verification

Local package is not the tester beta path.

Operators can still verify the package locally. Beta testers should use hosted HTTP or the Claude/Codex skill. 

python3 -m venv .venv
source .venv/bin/activate
pip install "git+https://github.com/arijitchowdhury80/scout.git@codex/scout-saas-prod-ready"
playwright install chromium
export SCOUT_WORKDIR="$PWD/scout-runs"
scout serve

Workdir

`SCOUT_WORKDIR` is where local Scout writes records, source pages, blocked pages, validation, reports, raw evidence, and screenshots.

Local-only health check

Only after `scout serve` is running on your own machine, open `http://localhost:8421/health` from that same machine.

Package release

`pip install scout-web` comes after registry publishing gates close. Until then, package installs remain operator verification only.

Skill boundary

The skill is the agent-facing wrapper. Hosted HTTP remains the acquisition surface.

Use small first runs

Start with one public URL, inspect records and evidence, then expand within your hosted credits.

First useful run

Make one artifact folder, then inspect it.

1. Pick a target

Start with a public page you are allowed to access.

2. Run Scout

Use hosted HTTP or the Claude/Codex skill. Keep live hard-site testing small and intentional.

3. Inspect evidence

Open `records.json`, `source_pages.json`, `blocked_pages.json`, and `extraction_report.md`.

Examples

Run small, inspectable workflows first.

Page to markdown

Capture one public page as readable text with source metadata.

mkdir -p ./scout-runs/example-page
scout scrape https://example.com \
  > ./scout-runs/example-page/scrape.json

Product category to records

Create product records plus blocked-page evidence for a small listing run.

scout products books \
  --site books.toscrape.com \
  --start-url https://books.toscrape.com/ \
  --output-dir ./scout-runs/books-products

Company intelligence packet

Produce a compact company packet with overview, key URLs, and citable source pages.

scout run company \
  --query Adobe \
  --mode auto \
  --workdir ./scout-runs/adobe-company

Artifact contract

Every useful run should leave evidence behind.

records.json

Structured records created from the run.

source_pages.json

Source registry with URLs, providers, timestamps, and hashes.

blocked_pages.json

Blocked, sparse, or failed pages with provider attempts and reasons where available.

extraction_report.md

A human-readable summary of what worked, what failed, and what should be reviewed next.

Readiness check

Verify the same launch gate CI runs.

The readiness checker summarizes the current truth: the controlled beta release is ready with limits. Public self-serve SaaS, paid checkout, public registries, and security-clean claims remain future gates. 

scout launch-readiness
scout launch-readiness --json
scout launch-readiness --require-public
scout launch-readiness --require-hosted-saas

Beta gate

The default command should report `Private beta: ready_with_limits` when the repo evidence is intact.

Release gate

`--require-public` should pass for the current controlled beta release.

Hosted SaaS gate

`--require-hosted-saas` should fail until SMTP delivery and Stripe checkout/webhook smoke are complete.
